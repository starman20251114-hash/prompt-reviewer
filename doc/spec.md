# プロンプトレビュアー 仕様書

## コンセプト

システムプロンプトを、固定のテストケースに対して繰り返し改善するツール。
テスト駆動開発のようにプロンプトを最適化することを目的とする。

---

## プロジェクト設定

- 対象LLMモデル / temperature 等のパラメータ（変更可・反映確認必須）
- APIキー（任意。未設定時はLLM機能を無効化し、手動スコアのみで動作）

---

## テストケース

### フィールド
| フィールド | 型 | 説明 |
|---|---|---|
| turns | `[{role, content}]` | マルチターン対応の会話履歴 |
| context_refs | `string[]` | コンテキスト参照（ファイルパスまたはテキスト） |
| expected_description | `string?` | 期待する出力の自由記述（任意） |

### コンテキスト参照
- アプリ内にViewer / 簡易Editorを内蔵
- 対象はテキスト/コードファイルを基本とする
- プロンプトやメッセージ内に `{{context}}` プレースホルダーで埋め込む
- プレースホルダーは1種類（`{{context}}`）

---

## プロンプトバージョン管理

| フィールド | 説明 |
|---|---|
| version | 自動採番 |
| name | 任意の名前 |
| memo | 任意のメモ |
| content | システムプロンプト本文 |
| parent_version_id | 分岐元バージョンID（任意） |
| created_at | 作成日時 |

- 任意のバージョンを起点に分岐して新バージョンを作成可能
- 破棄しない限りすべてのバージョンを保持・参照可能

---

## 実行（Run）

### モード
| モード | 説明 |
|---|---|
| 人間入力モード | 会話中にユーザーがリアルタイムで入力（メイン） |
| ダミーモード | LLMがユーザー役を演じる（動作確認用の完全ダミー） |

### ダミーモードの終了条件
1. LLMが「これ以上質問しない」と判断した場合
2. 10ターンを超えた場合 → システムが強制的に最終回答を要求

### Run のデータ構造
| フィールド | 説明 |
|---|---|
| prompt_version_id | 使用したプロンプトバージョン |
| test_case_id | 対象テストケース |
| conversation | 実行時の全会話 `[{role, content}]` |
| is_best | ベスト回答フラグ（バージョン×ケースごとに1件） |
| created_at | 実行日時 |

- 同一バージョン × 同一テストケースの複数回実行を保持
- ベスト回答は別枠でフラグ管理

---

## スコアリング

### 人間スコア
| フィールド | 説明 |
|---|---|
| human_score | 1〜5点 |
| human_comment | コメント（任意） |
| is_discarded | 破棄フラグ |

### 採点UI
- **個別採点**：1件ずつ応答を見ながら採点
- **一括採点**：全件を並べて採点、その場で破棄も可能

### LLM Judge（フェーズ2）
| フィールド | 説明 |
|---|---|
| judge_score | 1〜5点 |
| judge_reason | 採点理由 |

- 汎用プロンプトを内蔵（カスタマイズは将来検討）
- APIキー未設定時は無効

### 集計
- バージョンごとに全テストケースの平均スコアを算出して比較

---

## データ永続化

- **ローカル**：SQLite（スタンドアロンで完結）
- **クラウド**：PostgreSQL（接続先の切り替えのみで対応）
- Prisma を使用してDB種別を抽象化

### エクスポート/インポート
- フェーズ1：コピペ対応
- フェーズ3以降：CSV/JSON形式を検討

---

## 実装フェーズ

| フェーズ | 内容 |
|---|---|
| 1 | プロジェクト/テストケース/プロンプト管理 + 手動スコア（LLM不要） |
| 2 | LLM接続（Run実行 / Judge自動採点 / Improve提案） |
| 3 | ダミーモード / コンテキスト参照Viewer・Editor / エクスポート |

## 技術スタック

### 設計原則

**UI・ロジック・ストレージを分離**し、ホスト環境だけを差し替えられる構成にする。

```
[React + Vite SPA]      ← UI（どこでも動く純粋なSPA）
        ↕ API
[Hono]                  ← サーバー層（実行環境を選ばない）
        ↕
[Drizzle ORM]           ← DBアダプタ（SQLite / D1 / PostgreSQL）
```

### 各技術の選定理由

| 技術 | 理由 |
|---|---|
| **React + Vite SPA** | SSR不要のSPAにすることでTauri/拡張機能/Webすべてに対応 |
| **Hono** | Node.js・Bun・Cloudflare Workers・Denoで同じコードが動く |
| **Drizzle ORM** | SQLite（ローカル）/ D1（Cloudflare）/ PostgreSQL を差し替え可能 |

### デプロイターゲット別マッピング

| ターゲット | UI | サーバー | DB |
|---|---|---|---|
| **ローカル（サーバーなし）** | Viteビルド済みHTML | Hono on Node.js（自動起動） | SQLite |
| **Tauri（デスクトップ）** | Tauri内蔵WebView | HonoをSidecarとして起動 | SQLite |
| **Webアプリ** | 静的ホスティング | Hono on Node.js / Cloudflare Workers | PostgreSQL / D1 |
| **ブラウザ拡張** | Chrome Sidepanel | 不要（拡張内で完結） | IndexedDB |

### ディレクトリ構成（案）

```
prompt-reviewer/
├── packages/
│   ├── core/       # ビジネスロジック（フレームワーク非依存）
│   ├── server/     # Hono API（環境アダプタをここで切り替え）
│   ├── ui/         # React + Vite SPA
│   ├── desktop/    # Tauri設定
│   └── extension/  # Chrome拡張設定
```
